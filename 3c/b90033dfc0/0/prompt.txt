## Root Cause Analysis: Mobile Scroll Bug on Landing Page

### Files Involved

1. `/Users/fieldbradley/Projects/recoverylm/src/assets/styles/main.css` (lines 15-21) -- **primary culprit**
2. `/Users/fieldbradley/Projects/recoverylm/index.html` (line 6) -- contributing factor in PWA context
3. `/Users/fieldbradley/Projects/recoverylm/vite.config.ts` (line 32) -- PWA `standalone` mode amplifies the problem

### The Problem

The root cause is in `main.css` at lines 15-21:

```css
html, body {
  height: 100%;
}

#app {
  min-height: 100%;
}
```

Setting `height: 100%` on both `html` and `body` constrains the document to exactly the viewport height. On desktop browsers, this is generally harmless because the content naturally overflows and the browser still allows scrolling. However, on **mobile Safari and mobile Chrome** (especially in PWA standalone mode), this creates a scroll-locking effect:

1. The `body` is told it is exactly 100% of the viewport tall.
2. The landing page content inside `#app` is significantly taller than the viewport (hero, features, framework, privacy, disclaimer, and footer sections).
3. Mobile browsers interpret the fixed-height `body` as defining the scrollable boundary. When the content overflows a `height: 100%` body, mobile touch scrolling becomes unreliable -- the browser's scroll gesture detection gets confused because the "document" height and the "content" height disagree.
4. The PWA manifest sets `display: 'standalone'` and `orientation: 'portrait'`, which removes the browser chrome. In standalone mode, iOS Safari uses different scroll mechanics, and the `height: 100%` constraint becomes even more problematic.

This explains the exact symptoms described:
- **Forceful swipe required**: The touch scroll gesture needs to overcome the browser's confusion about whether the body is scrollable.
- **Rotating the phone fixes it**: When orientation changes, the browser recalculates the viewport and layout, which temporarily resolves the height constraint conflict. When rotated back, the recalculation persists until the next layout invalidation.

The `MainLayout.vue` uses `h-screen` (which is `height: 100vh`) and `overflow-hidden` intentionally for the app shell pages (chat, dashboard, etc.) where the individual pages manage their own scrolling. That is correct for those pages. But the landing page is rendered **outside** `MainLayout.vue` (it's a top-level route), so the global `html, body { height: 100% }` is the constraint that matters.

### Recommended Fix

**In `/Users/fieldbradley/Projects/recoverylm/src/assets/styles/main.css`**, change the `html, body` rule to use `min-height` instead of `height`, and keep the `#app` rule as-is:

```css
html, body {
  min-height: 100%;
}
```

This single change ensures:
- The `html` and `body` elements are **at least** viewport height (so short pages like unlock/setup still fill the screen with the dark background).
- When content is taller than the viewport (like the landing page), the body naturally expands to fit, and mobile touch scrolling works correctly.
- The app shell pages under `MainLayout.vue` are unaffected because they use explicit `h-screen` and `overflow-hidden` on their own wrapper elements, which independently constrains their height.

Alternatively, if there is concern about edge cases, a more targeted approach would be to keep `height: 100%` but add `overflow-y: auto` to `body`. However, `min-height: 100%` is the cleaner and more standard solution -- it is the universally recommended approach for mobile-friendly full-height layouts.

No changes are needed to the landing page component itself (`LandingPage.vue`), the `index.html`, or the `App.vue`. The issue is purely in the global CSS constraint.

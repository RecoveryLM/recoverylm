# Improvement

## Nonconformity and Corrective Action

**Purpose:** When something goes wrong, fix it — and fix the system that let it happen.

When a nonconformity occurs (a control fails, a process isn't followed, an incident happens), respond immediately to contain and correct it. Then dig into root causes: why did this happen, and could similar issues exist elsewhere? Implement corrective action, verify it worked, and update the governance system if needed.

For RecoveryLM, this could be triggered by: a crisis going undetected, an AI response that causes distress, a privacy breach, an audit finding, or a governance process that wasn't followed. The goal isn't blame — it's learning. Every incident is an opportunity to make the system more resilient. Document the nonconformity, what was done about it, and whether the corrective action was effective.

## Continual Improvement

**Purpose:** The governance system should get better over time, not just maintain the status quo.

Use everything available — monitoring results, audit findings, management review outputs, user feedback, incident learnings, changes in technology and best practices — to identify opportunities for improvement. For RecoveryLM, this means the governance system evolves alongside the product: as we learn more about how users interact with Remmi, as AI safety research advances, as the regulatory landscape shifts, and as our own understanding deepens through operating the system.

Governance that stands still while the world moves is governance that fails. Build improvement into the rhythm of the project.
